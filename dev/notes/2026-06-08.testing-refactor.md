# Replace `tox` with Poe + UV Task System

## Current State

- `tox.ini` runs pytest (with doctest) across Python 3.12, 3.13, 3.14, plus separate lint/fix environments
- `splatlog/poe.py` has a basic task definition for `splatlog:report`
- `pyproject.toml` has `requires-python = ">=3.12"` and already includes `poethepoet>=0.40.0`, `ruff>=0.15.0`, `pyright>=1.1.408`

## Operations


| Name        | Shorthand | Tool    | Check Command                        | Fix Command                  |
| ----------- | --------- | ------- | ------------------------------------ | ---------------------------- |
| `pytest`    | p, py     | pytest  | `pytest {targets}`                   | N/A                          |
| `doctest`   | d, doc    | pytest  | `pytest --doctest-modules {targets}` | N/A                          |
| `lint`      | l         | ruff    | `ruff check {targets}`               | `ruff check --fix {targets}` |
| `format`    | f, fmt    | ruff    | `ruff format --check {targets}`      | `ruff format {targets}`      |
| `typecheck` | t, tc     | pyright | `pyright {targets}`                  | N/A                          |


## Interface

The main interface will be a `test` task that runs all the operations.

```sh
poe test   # ->  lint, format, typecheck, pytest, doctest
```

The operation set can be adjusted with flags:

```sh
poe test --only pytest,doctest  # -> pytest, doctest
poe test --except typecheck     # -> lint, format, pytest, doctest
```

Shorthand operation names are supported, per the *Shorthand* column in the
*Operations* table:

```sh
poe test --only p,d  # -> pytest, doctest
poe test --except tc     # -> lint, format, pytest, doctest
```

### Fixing

Enable fixing with the `-f|--fix` flag:

```sh
poe test --fix
```

**Order matters here!** We want to apply the fix changes *before* type-checking
and running tests. In the case of a `-f|--fix` flag operations with a *Fix
Command* need to be performed first and sequentially.

> ⁉️ What should come first, `ruff check` or `ruff format`? Does it matter?
> Can they influence with each other?

### Targeting

#### Default Targets

The form of the `test` task command will be

```sh
poe test [OPTIONS] [TARGET...]
```

where targets are specified with the zero-or-more positional `TARGET` arguments.
When no positional arguments are given, each operation uses the first found of:

1.  Default target(s) configured for that operation.

    Per-operation config is important because running regular `pytest` on
    [tests](../../tests) and `--doctest-modules` on the source directory
    ([splatlog](../../splatlog) in this project) alleviates complaints about
    "test" in source file or directory names, as well as presumably speeding
    things up a tad.

2.  Default target(s) configured for all operations.

    > ⁉️ Will this actually be used/useful? What is the use case?

3.  No target arguments (operation command default).

#### Call-Time Targets

Each given `TARGET` argument is one of:

1.  File — include only this file
2.  Directory — include all file in this directory (and subdirectories)
3.  Git revision or range spec as the value of a `--git` option (`--git HEAD`,
    `--git main..HEAD`, see [gitrevisions][]) — include files changed (in single
    ref versus its parent(s), or between a range of refs).
  
    > ⁉️ Will this actually be used/useful, or is this just unnecessarily
    > complicating things? What is the use case, reducing the files considered
    > when run reactively?

### Optionality

Wish-list for options:

1.  Operation options can be configured or provided at call-time, with call-time
    taking precedence (call overrides config).
2.  Options can be global or operation-specific, with operation-specific
    taking precedence (specific overrides general).
3.  Support for additional configs to switch to or overlay in order to enable
    particular functionality, such as enabling logging and print output in
    `pytest` with something like:

    ```ini
    addopts = "--capture=no"
    log_cli = true
    log_cli_level = 20
    ```

#### Option-Parsing Compatibility

The first question that comes to mind is option-parsing compatibility between
the underlying operation commands. A quick investigation indicates quite high
compatibility, with all three tools appearing to support both _whitespace_ (WS)
separated options, like `--opt value`, and _equals_ (EQ) separated options, like
`--opt=value`. The form shown in the tool's `--help` display is indicated in the
_Help_ column.

| Tool    | Lang.      | Parse                 | Help | EQ? | WS? |
| ------- | ---------- | --------------------- | ---- | --- | --- |
| pytest  | Python     | argparse              | EQ   | Y   | Y   |
| ruff    | Rust       | clap                  | WS   | Y   | Y   |
| pyright | TypeScript | [command-line-args][] | WS   | Y   | Y   |

[command-line-args]: https://www.npmjs.com/package/command-line-args

#### Overriding Options

Good option-parsing compatibility between tools indicates we can structure
options as a key/value map with the usual methodologies for overriding, merging,
etc. and use that structure to generate option arguments that all tools will
understand.

#### Global and Specific Options

Though all operation commands may accept the same option forms, they certainly
don't accept the same sets of options. There may be a few cases where an option
can be passed to all underlying commands, but not many.

Operation-specific configuration is simple, as each operation can have its own
section:

```toml
[pytest.options]
ignore = "splatlog"

[doctest.options]
ignore = "tests"
```

We can support operation-specific options on the command line by prefixing the
option name with the operation name or shorthand:

```sh
poe test -- --pytest:ignore=splatlog --doctest:ignore=tests
poe test -- --py:ignore=splatlog --doc:ignore=tests
poe test -- --p:ignore=splatlog --d:ignore=tests
```

> 📝 IMPL These operation-specific options will not be declared in the `test`
> task args (way too many possibilities), so they will be passed as _free_ or
> _extra_ options, necessitating `--` before them (unless the `test` task
> declares _no_ args). Assuming that `test` is a _script_ task it looks like
> they will be available via `sys.argv` for parsing.

### Python Versions

We want to support simultaneous testing across multiple Python versions,
leveraging `uv` to provide the runtime and isolation.

> IMPL Poe supports runtime selection and isolation via the `executor`
> configuration on a task:
> 
> ```toml
> executor = { isolated = true, python = "3.12" }
> ```
> 
> We should be able to achieve it as well, using their facilities or mirroring
> it ourselves. Poe's [Tox Replacement Guide][] has links and details.
> 
> [Tox Replacement Guide]: https://poethepoet.natn.io/guides/tox_replacement_guide.html

#### Default Python Version Set

-   All _supported_ Python versions (_released_ but not yet _End-Of-Life (EOL)_)
    -   Equivalently, versions with _bugfix_ and _security_ statuses
-   Restricted by `pyproject.toml:project.requires-python`, if any
-   Version status from <https://peps.python.org/api/release-cycle.json>
    -   Data included in tree and refreshed at publish
    -   Periodically refreshed and stored locally on task runs (configurable)

#### Configured Python Version Set

Configuration overrides default set.

1.  Accepts a static list:

    ```toml
    python = ["3.10", "3.11", "3.12"]
    ```

2.  Accepts version statuses `prerelease`, `bugfix`, `security`, and
    `end-of-life`, in addition to the following aliases:

    | Name      | Replacement      |
    | --------- | ---------------- |
    | supported | bugfix, security |
    | sup       | supported        |
    | pre       | prerelease       |
    | bug       | bugfix           |
    | sec       | security         |
    | eol       | end-of-life      |

    For example, this setting would use all _supported_ (_bugfix_ and
    _security_) versions, plus the _prerelease_ version:

    ```toml
    python = ["supported", "prerelease"]
    ```

3.  Can mix-and-match, which is useful for including old versions:

    ```toml
    python = ["3.8", "3.9", "supported"]
    ```

#### Python Version Options

The version set can be specified per-task with the `-p|--python` command line
option:

```sh
poe test --python 3.12              # Single version

poe test --python 3.10,3.11,3.12    # Explicit list
poe test -p 3.10 -p 3.11 -p 3.12    # Same as above

poe test --python sup,pre           # Supported and prerelease versions
poe test -p 3.8,3.9,sup             # 3.8, 3.9, and all supported versions
```

Can also exclude versions with the `--no-python` option:

```sh
poe test --no-python 3.12           # Configured versions _except_ 3.12
poe test --no-python 3.10,3.11,3.12 # Exclude multiple versions
```

## 5. Result Aggregation for Parallel Runs

When running across multiple Python versions in parallel, aggregate results and
deduplicate errors.

```python
class ResultAggregator:
    def aggregate(self, results: dict[str, RunResult]) -> AggregatedResult:
        """
        Group identical errors across versions.
        
        Output format:
        - "All versions passed"
        - "Failed on 3.12, 3.13 (same errors):" followed by error once
        - "Different errors:" followed by version-specific errors
        """
        ...
```

### Implementation Approach

1. Capture stdout/stderr from each version run
2. Hash error output to detect duplicates
3. Present grouped summary

Note: This requires a **script task** (Python code) rather than a simple **cmd task** to handle the aggregation logic.
